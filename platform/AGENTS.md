# AGENTS.md

## 目标

一句话说明这个项目要完成什么。

## 目录

项目根目录只保留稳定结构，单次剪辑产生的过程文件放进当日工作文件夹。

- `AGENTS.md`：项目长期上下文，记录目标、目录、流程、输出和验收标准。
- `assets/`：提前准备好的产品素材、场景素材、人物素材、AI 片段和其他可复用素材。剪辑时只从这里复制素材，不移动或破坏原文件。
- `work/`：正在制作或待审核的视频项目。每次剪辑在这里创建一个当日工作文件夹。
- `final/`：已经完成并确认通过的视频，从 `work/` 迁移到这里，避免工作区文件过多。

每次剪辑时，在 `work/` 下创建一个当日工作文件夹：

```text
work/
  YYYY-MM-DD/
    reference-YYYY-MM-DD.mp4
    video_clips/
      reference-metadata.json
      reference-audio-16k.wav
      raw-transcript-whisper.txt
      reference-transcript.txt
      reference-copy-analysis.md
    script_sources.md
    script.txt
    titles.json
    storyboard/
      storyboard.json
      prompts/
      images/
    material/
      fragment01/
      fragment02/
    voice/
    jianying_draft/
```

工作台从参考链接创建任务时，书名尚未识别前使用
`YYYY-MM-DD-待确认书名-短ID/` 作为临时目录。口播分析得到最高置信度书名候选后，必须立即自动迁移为
`YYYY-MM-DD-书名/`；用户确认时如果纠正书名，目录必须再次同步更正。同一天存在同名任务时才依次追加
`-02`、`-03`。自动命名不得替代人工书名确认门。迁移目录时必须同步更新任务索引和产物路径，不得继续使用平台名或任务 ID 作为正式项目目录名。

当日工作文件夹说明：

- `reference-YYYY-MM-DD.mp4`：使用 TikHub 从用户提供的抖音链接下载的完整参考视频，只作为内容结构、开头方式、情绪曲线、镜头密度、字幕样式和节奏参考。若同一天有多个任务，可在日期后追加短标题或序号。
- `video_clips/reference-metadata.json`：TikHub 返回的来源链接、抖音作品 ID、标题、作者、互动数据、下载时间、本地路径、文件大小和 SHA-256。
- `video_clips/raw-transcript-whisper.txt`：Codex 使用本机 Whisper 得到的原始逐字稿，必须在清洗前立即落盘并保留供审计。
- `video_clips/reference-transcript.txt`：原始逐字稿繁转简并经过最小校对后的参考文案，只允许修正有上下文依据的同音错字、错分词、标点和分段，不得改写。
- `video_clips/reference-copy-analysis.md`：使用 `dbs-content` 对参考文案的爆点、钩子、内容结构、情绪推进、语言节奏、结尾方式、可迁移抽象框架和不可照搬元素所做的诊断。
- `video_clips/`：除上述证据外，还保存参考视频的画面片段和参考帧，用于分析结构与包装方式，不直接决定新视频的逐镜头内容。
- `script_sources.md`：G01 来源包，记录 TikHub 证据、参考文案、DBS 诊断、微信读书版本、热门划线、引用边界和未核验表达。
- `script.txt`：在 G01 来源包得到用户确认后，融合获批抽象框架、微信读书热门划线与原创感悟生成，并经用户审核的二创口播稿。它是分镜、背景图、配音和字幕的内容源头。
- `titles.json`：在 G02 文案确认后生成的标题选择记录。必须保存抖音原标题、10 个长标题候选、每个候选的 `dbs-xhs-title` 公式编号/类型/模板/原始爆款例子/推荐理由、已确认长标题、10 个短标题候选和已确认短标题。
- `storyboard/storyboard.json`：根据已审核文案生成的分镜表，记录每个镜头对应文案、画面描述、时长、景别、构图、人物连续性和图片路径。
  - 每个 `storyboard.json` 必须包含机器可读的 `segmentation_rules`。不超过约 60 秒的口播默认生成 7–8 张图（包含 G03 风格样图），优先 8 张，约 8 秒一张，6–10 秒为常用软范围。
  - 分镜数量不是机械硬指标：必须先按观点、动作、场景、情绪和叙事功能的变化切分，再用时长校正。不得切断完整因果句或转折句，不得为了凑数量制造语义重复、构图重复或无独立叙事价值的镜头。
  - 配音前使用文案语义和字数估时；配音后以真实语音时长为准，优先调整画面停留时长。只有单镜头明显过长且存在自然语义边界时才新增镜头，不能擅自破坏已确认的文案含义。
- `storyboard/prompts/`：每个分镜的背景图生成提示词和统一风格设定。
- `storyboard/images/`：根据分镜提示词生成并经审核的竖屏背景图。背景图默认不含书名、金句、字幕或水印，文字在剪映文本轨道中添加。
- `material/fragmentNN/`：从 `assets/` 筛选并复制过来的当前片段候选素材或已批准素材。
- `voice/`：按片段生成的配音、清理后人声、分段语音和拼接后的成品音频。
- `jianying_draft/`：当前样稿对应的剪映草稿、草稿检查报告和必要备份。只在需要生成剪映草稿时创建。

关键来源规则：

- 完整参考视频及其逐字稿只提供钩子类型、内容结构、信息顺序、情绪曲线、语言节奏、结尾功能和包装参考；不得直接作为新视频文案、音频或逐镜头画面的生产依据。
- 参考视频文案与微信读书是两类独立证据。参考视频中的句子未经微信读书独立核验，不得标成书中原句。
- `reference-copy-analysis.md` 只提炼可迁移的抽象机制，不得把参考视频的独特措辞、例子、主张或人物设定当作二创素材。
- `script_sources.md` 是 G01 的来源证据与引用边界；用户未明确确认来源包前，不得生成 `script.txt`。
- 经用户确认的 `script.txt` 是新视频内容、分镜、配音和字幕的首要依据；文案未确认前不批量生成背景图。
- `storyboard/storyboard.json` 是视觉顺序、画面语义和背景图生成的机器可读依据。
- `storyboard/images/` 与 `material/fragmentNN/` 是审校通过后的生产素材来源。
- 使用生成配音时，以每段真实完成语音时长作为画面和字幕时长依据。
- `recipe.json` 记录最终镜头时间轴；`storyboard/storyboard.json` 记录文案到画面的映射；使用素材库替换或补充 AI 背景图时，再生成 `matches.json`。

默认生产变体：

- 未指定旁白性别时，直接使用女声版本，不新增旁白性别确认门。
- 女声版本固定配对 `female-book-narrator-locked-v1` 与 `固定/女版前3秒固定开头.mp4`。
- 只有用户明确要求男声时，才切换男声音色和男版固定片头；不得从旧项目、遗留配置、文件名或历史偏好推断男声。
- 音色与固定片头必须同变体配对，未经用户明确授权不得交叉使用。

## 流程

### 强制确认顺序（不得跳步）

用户提供抖音参考链接、书名或制作说明后，必须严格按以下顺序执行；每个确认点都是阻塞门，未获得用户明确确认时必须停止，不得提前执行下一阶段：

1. **生成并确认 G01 来源包**：
   - 使用 TikHub 下载抖音参考视频并保存来源元数据；必须配置 `TIKHUB_API_KEY`，TikHub 不可用时明确阻塞，不得静默改用 yt-dlp、浏览器抓取、模拟数据或其他下载源。
   - 使用 Codex/本机 Whisper 提取原始逐字稿，先保存 `raw-transcript-whisper.txt`，再生成只做繁转简与最小校对的 `reference-transcript.txt`。
   - 使用 `dbs-content` 诊断参考文案的爆点、钩子、内容承诺、冲突、结构、情绪曲线、语言节奏、结尾方式和可迁移抽象框架；`dbs-content` 只负责诊断，不负责代写二创文案。
   - 使用微信读书确认正确书名、作者、译者和版本，获取全书划线人数最多、热度最高的热门划线。
   - 向用户一次性展示参考文案、DBS 诊断、微信读书候选原句、章节、划线人数、引用边界和未核验表达，并停止等待明确确认。
2. **生成二创文案并确认**：仅在 G01 来源包获得用户明确确认后，保留获批的抽象框架，用获批微信读书观点和热门划线替换参考视频的措辞、例子和主张，再加入原创感悟。必须核对直接引用并对照参考逐字稿检查独特措辞和句序重合，交给用户审核。
3. **生成并确认长标题**：仅在 G02 文案确认后，读取 TikHub 获取的抖音原标题，使用 `dbs-xhs-title` 从 75 个公式中匹配 5–8 个适用公式（至少覆盖 3 类心理触发器），参照原标题的长度、口语节奏和情绪强度仿写 10 个视频号长标题。每个候选必须保留公式编号、模板、原始爆款例子和推荐理由；展示后停止，等待用户确认 1 个。
4. **生成并确认短标题**：仅在用户明确确认长标题后，基于该已选长标题生成 10 个 4–12 字为主、最长不超过 16 字的短标题；展示后停止，等待用户确认 1 个。重新生成或更换长标题时，必须清空短标题候选与确认状态。
5. **生成一张风格样图并确认**：只有长标题和短标题都明确确认后，才根据已确认文案完成分镜方向并生成一张有代表性的 9:16 风格样图。只有用户明确确认画风、色彩、人物和构图后，才能生成余下图片。
6. **生成余下图片**：沿用已确认样图的画风、色彩、人物身份、时代背景、光线和构图规则，生成其余分镜图片。不得在风格未确认时批量生图。
7. **继续后期制作**：全部图片审核通过后，才能继续配音、真实时长对齐、字幕、样稿、剪映草稿和终稿制作。

禁止把“用户只提供书名”视为对文案、风格或批量生图的默认批准。确认必须来自用户明确回复，不得用 Agent 自审代替用户确认，也不得并行或倒置以上阶段。

流程路径：

TikHub 下载抖音参考视频 → Codex/Whisper 提取并最小校对参考文案 → `dbs-content` 分析爆点、内容结构与框架 → 微信读书确认版本并获取热门划线 → 展示 G01 来源包 → 用户确认来源包 → 融合生成二创文案 → 用户确认文案 → `dbs-xhs-title` 生成 10 个长标题 → 用户确认长标题 → 基于已选长标题生成 10 个短标题 → 用户确认短标题 → 按文案生成分镜 → 生成一张风格样图 → 用户确认风格 → 生成余下图片 → 审核全部图片 → 生成配音 → 按真实配音时长生成时间轴 → 生成样稿和剪映草稿 → 审核修改 → 生成终稿

1. TikHub 下载参考视频：在 `work/YYYY-MM-DD-short-title/` 下创建当日工作文件夹，运行 `produce-wechat-book-video/scripts/download_douyin_tikhub.mjs`，将用户提供的抖音链接下载为 `reference-YYYY-MM-DD.mp4`，并把 TikHub 元数据保存为 `video_clips/reference-metadata.json`。不得覆盖已有参考视频；TikHub 失败时停止并暴露错误。
2. Codex 提取参考文案：提取 16k 单声道音频，使用本机 Whisper 完成 ASR，立即保存 `video_clips/raw-transcript-whisper.txt`；再经 OpenCC 繁转简和最小校对生成 `video_clips/reference-transcript.txt`。
3. DBS 诊断参考文案：使用 `dbs-content` 分析内容承诺、开头钩子、第一冲突、信息顺序、情绪推进、句式节奏、金句机制、结尾功能、可迁移抽象框架和不可照搬元素，保存为 `video_clips/reference-copy-analysis.md`。
4. 获取微信读书热门划线：确认正确图书版本，获取全书划线人数最多、热度最高的语句。若微信读书不可用，必须明确说明，未经用户同意不得改用其他来源替代。
5. 生成 G01 来源包：把 TikHub 证据、参考文案、DBS 诊断、微信读书版本、候选原句、章节、划线人数、引用边界和未核验表达写入 `script_sources.md`，向用户展示后停止等待确认。
6. 生成二创文案：仅在 G01 来源包获得用户明确确认后，保留获批抽象框架，用获批微信读书证据与原创感悟重建所有内容句，保存为 `script.txt`。必须对照 `reference-transcript.txt` 检查并改写独特措辞、连续句序、例子和未核验主张；明确区分直接引用与原创表达，并停止等待用户确认。
7. 生成长标题：读取 `reference-metadata.json` 中的抖音原标题，使用 `dbs-xhs-title` 匹配 5–8 个公式并生成 10 个长标题，保存到 `titles.json` 后停止等待用户选择。标题长度尽量贴近原标题，默认允许约 ±20% 的浮动；不得照抄原标题的独特措辞或连续句序。
8. 生成短标题：仅在长标题明确确认后，基于已选长标题生成 10 个短标题并写入 `titles.json`，停止等待用户选择。长短标题都确认前，工作台和服务端都必须阻止场景图生成。
9. 按文案生成分镜：仅在用户明确确认文案、长标题和短标题后，按语义和叙事节奏拆分 `script.txt`，生成 `storyboard/storyboard.json`，为每个镜头定义文案范围、画面内容、景别、构图、情绪、连续性要求和背景图提示词。不超过约 60 秒的视频默认切成 7–8 张图（包含后续 G03 样图），以约 8 秒一张作为软目标；语义完整、画面独立性和构图变化优先于机械时长。
10. 生成风格样图：只生成一张有代表性的 9:16 风格样图供用户确认，然后停止等待反馈；不得同时或提前生成余下图片。
11. 生成余下背景图：仅在用户明确确认风格样图后，为其余分镜生成 9:16 竖屏背景图并保存到 `storyboard/images/`。图片默认不包含文字和水印，书名、金句及字幕在可编辑文本轨道中完成。
12. 补充或替换素材：需要真实书封、作者、书页、人物或动态镜头时，从 `assets/` 筛选并复制素材到 `material/fragmentNN/`；缺素材必须明确暴露，不用无关内容硬凑。
13. 生成配音：默认使用本机 VoxCPM2 生成新配音，分段语音和合成音频保存到 `voice/`，并读取每段真实完成时长。
   - 本项目默认使用女声旁白，固定使用 `assets/voice-presets/female-book-narrator-locked-v1.json`，不得自行更换参考音色、音调、种子或高保真参数；只有用户明确要求男声时，才切换为男声版本。
   - 固定女声以 `female-book-narrator-locked-v1-reference.wav` 为音色基准，并使用 `female-book-narrator-locked-v1-transcript.txt` 中的原始提示词，采用 VoxCPM2 `prompt_and_reference` 模式；参数固定为 `cfg-value=2.0`、`inference-timesteps=20`、`seed=42`，禁止按分段递增或随机改变 seed。
   - 默认保持模型原生 `1.00x` 语速，不做时间伸缩；书名后停顿 `1.8s`，普通句间停顿 `0.5–0.6s`，结尾停顿 `0.3s`。只有用户明确提出新的速度要求时，才使用保音调、保共振峰变速。
   - 参考视频 ASR 默认使用本机 `whisper.cpp v1.9.1` 与多语言 `ggml-small.bin`，程序安装在 `F:\Codex\tools\whisper.cpp\v1.9.1\`，模型安装在 `F:\Codex\models\whisper.cpp\`。原始逐字稿必须在 ASR 完成后立即落盘，不得因后续 LLM 清洗失败而丢失。
   - Whisper 原始输出另存为 `raw-transcript-whisper.txt` 供审计；Codex 工作稿 `reference-transcript.txt` 必须先通过本地 OpenCC 转成简体中文，再逐句检查有明确上下文依据的书名同音字、近音错字、错分词、常用搭配、标点和断句，按语义转折分段、段间空一行。只允许最小校对，严禁借校对或分段之名改写、扩写或重排原视频内容。
14. 生成时间轴：根据真实配音时长更新 `recipe.json`，使每段画面、字幕和配音对齐；参考视频时长只作为节奏参考。
15. 生成样稿和剪映草稿：用已审核背景图、补充素材和新配音生成版本化样稿，并创建新剪映草稿保存到 `jianying_draft/`，供用户打开审核。
16. 生成独立视频号封面：核验原版书封后，按固定规格生成 1080x1260 独立封面，不嵌入视频。
17. 生成终稿：根据用户审核意见只修改有问题的文案、分镜、背景图、字幕或配音；用户明确确认成片、剪映草稿和独立封面后结束默认生产流程。发布与归档需要另行明确授权。

## 输出

保存通用输出配置；单次特殊要求以用户当前指令和 skill 执行结果为准。

- 视频比例：9:16 竖屏。
- 分辨率：1080x1920。
- 帧率：60fps。
- 文件命名规则：`YYMMDD-platform-product-序号.mp4`，英文缩写优先，避免文件名过长或乱码。
- 默认审核方式：生成剪映草稿，供用户打开审核。
- 默认归档方式：用户确认通过后，从 `work/` 迁移到 `final/`。
- AI 背景图规格：9:16 竖屏构图，优先生成 1080x1920 或更高分辨率；默认无文字、无标志、无水印，并为字幕与书名保留安全区。

## 验收

视频合格标准：

- 抖音参考视频必须由 TikHub 下载，`reference-metadata.json` 中的来源链接、作品 ID、本地路径、文件大小和 SHA-256 必须完整；TikHub 失败不得静默切换下载源。
- `raw-transcript-whisper.txt`、`reference-transcript.txt` 和 `reference-copy-analysis.md` 必须存在；原始逐字稿不得被最小校对稿覆盖。
- G01 来源包必须同时包含参考文案、DBS 诊断、微信读书版本与热门划线、引用边界和未核验表达；用户明确确认 G01 前不得生成 `script.txt`。
- 二创文案必须只继承获批抽象框架，并用微信读书证据与原创感悟重建内容句；除书名和独立核验的短引文外，不得保留参考视频的独特措辞、连续句序、例子或未核验主张。
- 长标题必须恰好提供 10 个候选并可追溯到 `dbs-xhs-title` 公式；确认长标题前不得生成短标题。短标题必须恰好提供 10 个候选，且只能基于已确认长标题生成；长短标题都确认前不得生成风格样图或其他场景图。
- 每个视觉片段都有可用素材；缺素材和低置信度片段必须明确暴露。
- 不用无关素材硬凑数量。
- 素材画面和产品、场景、动作、景别、角度、光线、产品可见度和构图基本匹配。
- 文件名和文件夹名只能作为线索，最终要看画面。
- 复制素材，不移动或破坏原始素材。
- 识别重复素材，不能把不同文件名当成不同内容。
- 三个连续镜头范围内尽量避免同一来源文件夹重复出现。
- 相邻片段避免明显重复的构图或语义场景。
- 有关联的 AI 片段或同一角色片段要保持身份连续。
- 批量生成 AI 背景图前必须先确认一张风格样图；同一视频的画风、色彩、人物身份、时代背景和光线逻辑保持连续。
- 每张背景图必须对应具体文案语义和叙事功能，不能只追求视觉漂亮而与口播无关。
- AI 背景图默认不生成中文书名、金句或字幕，避免错字和不可编辑文字；文字统一放在剪映文本轨道。
- 书名、作者、书中观点和直接引用必须核对；无法确认的内容改为概述，不伪造原句、页码或作者观点。
- 成片视觉速度保持 `1.00x`，除非用户明确要求变速；短素材可定格补足，不通过变速硬拉。
- 成片比例、分辨率和帧率符合本文件的通用配置；时长和数量符合用户当前要求或本次任务设置。
- 成片时长和参考或本次目标时长的偏差应在可接受范围内，明显偏差要说明。
- 字幕必须在所属片段内，按语义断句，不机械凑字数。
- 如本次任务设置字幕字数上限，应把它当作上限，不是目标；避免 1-2 个字的孤立字幕。
- 字幕不遮挡产品主体、关键动作或重要信息。
- 如果使用配音，以真实成品配音时长为画面和字幕依据。
- 配音要检查开头、中间边界、最长停顿、最紧字幕边界和结尾。
- 默认女声旁白必须与 `assets/voice-presets/female-book-narrator-locked-v1-full.wav` 的音色、音调和清晰度一致；如需修正误读，只重生成问题片段并使用同一 v1 参考音色、原始提示词、`prompt_and_reference` 模式与固定 seed。用户明确要求男声时，才按对应男声锁定预设及其验收标准执行。
- 如生成剪映草稿，必须是新建可编辑草稿，不能覆盖原生或加密缓存。
- 剪映草稿的内容 ID、元数据 ID 和根索引 ID 必须一致，批量草稿 ID 不能重复。
- 剪映草稿的素材路径必须存在，媒体源范围不能超过真实时长。
